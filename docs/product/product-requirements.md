# Product Requirements Document (PRD)

## Матрица компетенций — Matrix

**Версия:** 1.0
**Дата:** 2026-03-24
**Статус:** Черновик

---

## 1. Общие сведения

### 1.1 Назначение

Matrix — веб-приложение для управления матрицей компетенций сотрудников управления сопровождения портала динамической инфраструктуры Сбер. Система обеспечивает прозрачную оценку навыков, выявление зон роста, планирование карьерных треков и формирование индивидуальных планов развития.

### 1.2 Scope

Система охватывает полный цикл управления компетенциями:

- Ведение каталога компетенций (общие и специфичные по отделам)
- Оценка 360° (самооценка, коллеги, тимлид, руководитель)
- Аналитика и визуализация (радар, тепловые карты, gap-анализ)
- Карьерные треки и планы индивидуального развития
- Экспорт отчётов и интеграция с Telegram

### 1.3 Целевая аудитория

Управление сопровождения портала динамической инфраструктуры — 5 отделов, от 10 до 50 сотрудников.

**Отделы (по возрастанию уровня экспертизы):**

| № | Отдел | Профиль |
|---|-------|---------|
| 1 | Первая линия сопровождения | Базовая поддержка, квоты, известные ошибки |
| 2 | Дежурная смена | Troubleshooting, Ansible, vCloud Director, инциденты |
| 3 | Бизнес-логика | Верификация бизнес-логики, выделение ресурсов |
| 4 | Вторая линия (SRE/DevOps) | Логи, метрики, код, базы данных |
| 5 | Jenkins CDP | Автоматизация Jenkins |

**Роли пользователей:**

| Роль | Описание | Область видимости |
|------|----------|-------------------|
| Admin | Полный доступ ко всем функциям | Вся система |
| Department Head | Руководитель отдела/управления | Все отделы и сотрудники |
| Team Lead (TL) | Тимлид команды | Своя команда |
| HR | HR-специалист | Отчёты и аналитика по всем |
| Employee | Сотрудник | Свой профиль |

### 1.4 Глоссарий

| Термин | Определение |
|--------|-------------|
| Компетенция | Измеримый навык или знание, необходимое для выполнения рабочих задач |
| Целевой профиль | Набор компетенций и их целевых уровней для конкретной роли/должности |
| Кампания оценки | Организованный цикл оценки 360°, проводимый раз в 6 месяцев |
| Gap-анализ | Анализ разрыва между текущим и целевым уровнем компетенций |
| IDP | Individual Development Plan — индивидуальный план развития |
| Карьерный трек | Маршрут развития сотрудника между отделами/должностями |

## 2. Функциональные требования

### 2.1 Модуль "Управление организацией"

#### FR-001: CRUD сотрудников

Система должна обеспечивать создание, чтение, обновление и удаление карточек сотрудников.

**Атрибуты сотрудника:**
- ФИО (фамилия, имя, отчество)
- Должность
- Отдел (из справочника отделов)
- Команда (из справочника команд)
- Email (уникальный, используется для логина)
- Telegram username (для уведомлений)
- Дата приёма на работу
- Статус (активный / уволен)

**Правила:**
- Email уникален в системе
- При удалении сотрудника его оценки и история сохраняются (soft delete)
- Только Admin и Head могут создавать/удалять сотрудников
- TL может редактировать сотрудников своей команды

#### FR-002: CRUD отделов

Система должна поддерживать 5 предустановленных отделов с возможностью добавления новых.

**Атрибуты отдела:**
- Название
- Описание
- Руководитель (ссылка на сотрудника)
- Порядок сортировки (определяет уровень в иерархии)

**Правила:**
- Предустановленные отделы нельзя удалить, только переименовать
- Удаление отдела возможно только при отсутствии привязанных сотрудников
- Только Admin может управлять отделами

#### FR-003: CRUD команд внутри отделов

Система должна поддерживать произвольное количество команд внутри каждого отдела.

**Атрибуты команды:**
- Название
- Отдел (родительский)
- Тимлид (ссылка на сотрудника)
- Описание

**Правила:**
- Команда принадлежит ровно одному отделу
- У команды может быть один тимлид
- Admin и Head могут управлять командами

#### FR-004: Назначение ролей

Система должна позволять назначать роли пользователям.

**Роли и их иерархия:**
1. **Admin** — полный доступ; может назначать любые роли
2. **Department Head** — видит все отделы, управляет сотрудниками и оценками
3. **Team Lead** — видит свою команду, управляет оценками внутри команды
4. **HR** — доступ к отчётам и аналитике (read-only для персональных данных)
5. **Employee** — доступ к своему профилю и своим оценкам

**Правила:**
- Сотрудник может иметь одну роль
- Только Admin может назначать/менять роли
- При назначении TL автоматически привязывается к команде

#### FR-005: Импорт сотрудников из CSV/Excel

Система должна поддерживать массовый импорт сотрудников.

**Поддерживаемые форматы:** CSV (UTF-8), XLSX

**Обязательные колонки:** ФИО, email, отдел, должность

**Правила:**
- Перед импортом — предварительный просмотр данных и валидация
- Ошибки валидации отображаются построчно (номер строки + текст ошибки)
- Дубликаты по email — обновляют существующие записи (с подтверждением)
- Импорт доступен только Admin и HR

### 2.2 Модуль "Каталог компетенций"

#### FR-010: CRUD компетенций с категориями

Система должна обеспечивать ведение каталога компетенций.

**Атрибуты компетенции:**
- Название
- Описание
- Категория: Hard Skills | Soft Skills | Processes | Domain Knowledge
- Тип: общая (common) | специфичная (department-specific)
- Привязка к отделам (если специфичная)
- Статус: активна | архивирована

**Правила:**
- Архивированная компетенция не участвует в новых оценках, но сохраняется в истории
- Только Admin и Head могут управлять каталогом

#### FR-011: Общие и специфичные компетенции

Система должна разделять компетенции на общие и привязанные к отделам.

**Общие компетенции (примеры):**
- Коммуникация (Soft)
- Работа в команде (Soft)
- Управление инцидентами (Process)
- Документирование (Process)

**Специфичные компетенции (примеры):**
- Ansible (Hard) — Дежурная смена, SRE/DevOps
- vCloud Director (Hard) — Дежурная смена
- Jenkins Pipeline (Hard) — Jenkins CDP
- SQL/DB Administration (Hard) — Вторая линия

**Правила:**
- Общие компетенции видны при оценке любого сотрудника
- Специфичные компетенции видны только для сотрудников привязанного отдела
- Компетенцию можно привязать к нескольким отделам
- При просмотре кросс-отдельной аналитики видны все компетенции

#### FR-012: 5 уровней владения с описаниями

Каждая компетенция оценивается по шкале 0–4.

| Уровень | Название | Описание |
|---------|----------|----------|
| 0 | Отсутствует (None) | Навык не применялся, знания отсутствуют |
| 1 | Новичок (Novice) | Базовое понимание, работает под руководством, требуется наставник |
| 2 | Базовый (Basic) | Самостоятельно выполняет типовые задачи, обращается за помощью в сложных случаях |
| 3 | Продвинутый (Advanced) | Решает сложные задачи самостоятельно, может обучать других |
| 4 | Эксперт (Expert) | Глубокая экспертиза, определяет стандарты, менторит команду |

**Правила:**
- Для каждой компетенции Admin/Head может задать индивидуальные описания критериев по каждому уровню
- Если индивидуальные описания не заданы, используются описания по умолчанию (таблица выше)

#### FR-013: Обучающие ресурсы

Система должна позволять привязывать обучающие ресурсы к компетенциям.

**Атрибуты ресурса:**
- Название
- URL
- Описание
- Тип: курс | статья | видео | книга | внутренний документ
- Рекомендуемый уровень (для какого уровня компетенции полезен)

**Правила:**
- Один ресурс может быть привязан к нескольким компетенциям
- Ресурсы видны сотруднику при просмотре gap-анализа
- Любой пользователь может предложить ресурс; утверждение — Admin, Head или TL
- Удаление ресурсов: любой может запросить → подтверждение Admin/Head/TL (принцип «второй руки»)

#### FR-014: Целевые профили

Система должна поддерживать определение целевых профилей для ролей/должностей.

**Атрибуты целевого профиля:**
- Название (например, «Инженер дежурной смены — уровень Senior»)
- Отдел
- Должность / роль
- Набор компетенций с целевыми уровнями

**Правила:**
- Целевой профиль можно назначить сотруднику
- Один профиль может быть назначен нескольким сотрудникам
- При сравнении текущих оценок с целевым профилем формируется gap-анализ
- Руководитель управления и Admin могут редактировать целевые профили любого отдела
- Руководитель отдела может редактировать целевые профили своего отдела
- TL не может редактировать целевые профили, но может предлагать изменения (заявка → утверждение Admin/руководитель)

#### FR-015: Импорт компетенций из CSV/Excel

Система должна поддерживать массовый импорт компетенций.

**Поддерживаемые форматы:** CSV (UTF-8), XLSX

**Обязательные колонки:** название, категория

**Опциональные колонки:** описание, тип, отделы, описания уровней

**Правила:**
- Валидация категорий (только Hard/Soft/Processes/Domain)
- Предварительный просмотр перед импортом
- Доступно только Admin

### 2.3 Модуль "Оценка 360°"

#### FR-020: Создание кампании оценки

Система должна поддерживать создание и управление кампаниями оценки.

**Атрибуты кампании:**
- Название (например, «Оценка Q1 2026»)
- Дата начала
- Дата окончания (дедлайн)
- Участники (список сотрудников или выбор по отделам/командам)
- Метод агрегации (среднее / медиана / взвешенное)
- Описание / комментарий

**Правила:**
- Кампании создаёт Admin или Head
- Период кампании — рекомендуемый цикл 6 месяцев
- При создании кампании автоматически формируются задачи на оценку для участников

#### FR-021: Самооценка сотрудника

Сотрудник должен иметь возможность оценить себя по всем компетенциям, входящим в его целевой профиль (или по компетенциям отдела).

**Правила:**
- Самооценка обязательна для каждого участника кампании
- Оценка ведётся по шкале 0–4 для каждой компетенции
- Возможность добавить текстовый комментарий к каждой оценке
- Самооценку можно сохранить как черновик и вернуться позже
- После отправки самооценку нельзя изменить

#### FR-022: Оценка коллегами (peers)

Коллеги могут оценивать друг друга в рамках кампании.

**Правила:**
- Сотрудник сам выбирает рецензентов (peers) из списка коллег
- Минимум 1 peer-оценщик, максимум 5
- Рецензент не может отказаться от назначения
- Если рецензент не завершил оценку до дедлайна — автоматическое продление на 2 недели; после продления незавершённая оценка исключается из расчёта (вес перераспределяется)
- Peer-оценка анонимна для оцениваемого сотрудника (видны только агрегированные результаты)
- Оценщик видит только компетенции, релевантные оцениваемому
- Peer-оценки видны Admin, Head и TL (для контроля качества)

#### FR-023: Оценка тимлидом

Тимлид оценивает каждого сотрудника своей команды.

**Правила:**
- TL оценивает по всем компетенциям профиля сотрудника
- Оценка TL не анонимна
- TL может добавить развёрнутый текстовый комментарий
- Оценка TL обязательна для завершения кампании по сотруднику

#### FR-024: Оценка руководителем отдела/управления

Руководитель (Head) может оценить любого сотрудника управления.

**Правила:**
- Оценка Head опциональна (не блокирует завершение кампании)
- Head может оценивать сотрудников любого отдела
- Оценка не анонимна
- При настройке взвешенной агрегации оценке Head может быть присвоен повышенный вес

#### FR-025: Агрегация оценок

Система должна агрегировать оценки из всех источников.

**Методы агрегации (настраивается на уровне кампании):**
- **Среднее арифметическое** — (self + peers + TL + Head) / N
- **Медиана** — медианное значение всех оценок
- **Взвешенное среднее** — с настраиваемыми весами (по умолчанию):
  - Руководитель управления/отдела (Head): 0.35
  - Тимлид (TL): 0.30
  - Самооценка: 0.20
  - Peers (коллеги): 0.15
  - Формула: `final_score = head_score × 0.35 + tl_score × 0.30 + self_score × 0.20 + peer_avg × 0.15`

**Правила:**
- Агрегированная оценка рассчитывается автоматически при завершении кампании
- Агрегированная оценка округляется до одного десятичного знака
- Если какой-то источник отсутствует, веса перераспределяются пропорционально
- Полная история оценок сохраняется (все исходные оценки + агрегированный результат)

#### FR-026: Статусы кампании

Жизненный цикл кампании оценки:

```
Черновик → Активна → Сбор завершён → Калибровка → Финализирована → Архив
```

| Статус | Описание |
|--------|----------|
| Черновик | Кампания создана, настраивается. Оценки недоступны |
| Активна | Сотрудники могут проходить оценку. Рассылаются уведомления |
| Сбор завершён | Все оценки собраны (или дедлайн истёк). Подготовка к калибровке |
| Калибровка | Руководители проводят калибровочную сессию, разрешают расхождения |
| Финализирована | Результаты утверждены после калибровки. Изменения невозможны |
| Архив | Кампания перенесена в архив. Данные доступны для исторического анализа |

**Правила:**
- Переход Черновик → Активна: только если назначены участники
- Переход Активна → Сбор завершён: вручную Admin/Head или автоматически по дедлайну
- При автоматическом завершении сбора: незаполненные оценки помечаются как «не предоставлена»
- Переход Сбор завершён → Калибровка: вручную Admin/Head
- Переход Калибровка → Финализирована: вручную Admin/Head после утверждения результатов
- Переход Финализирована → Архив: вручную Admin

#### FR-027: Уведомления (Telegram + email)

Система должна отправлять уведомления через два канала: Telegram (основной) и email (резервный/альтернативный).

**Типы уведомлений:**
- Старт кампании: «Начата кампания оценки {название}. Дедлайн: {дата}»
- Напоминание: за 3 дня и за 1 день до дедлайна (если оценка не завершена)
- Завершение кампании: «Кампания {название} завершена. Результаты доступны»
- Назначение оценщиком: «Вам назначена оценка сотрудника {ФИО}»

**Категории уведомлений:** оценка, IDP, карьерные треки, системные

**Правила:**
- Основной канал — Telegram Bot API; если у сотрудника нет Telegram — fallback на email
- Пользователь может настроить типы уведомлений (вкл/выкл по категориям)
- Для отправки в Telegram необходим заполненный Telegram username в профиле
- Если ни Telegram, ни email недоступны — уведомление логируется без ошибки

### 2.4 Модуль "Аналитика и визуализация"

#### FR-030: Радарная диаграмма

Система должна отображать радарную (паутинную) диаграмму компетенций сотрудника.

**Функциональность:**
- Оси — компетенции из профиля сотрудника
- Линия 1 (сплошная) — текущий уровень (агрегированная оценка)
- Линия 2 (пунктирная) — целевой уровень из профиля
- Возможность наложения результатов разных кампаний для сравнения динамики
- Фильтрация по категории компетенций (Hard/Soft/Processes/Domain)

**Правила:**
- Радар доступен: Employee — свой профиль; TL — свою команду; Head/Admin — все; HR — все (read-only)
- Максимум 12 осей на одном радаре; при большем количестве — группировка по категориям

#### FR-031: Тепловая карта

Система должна отображать тепловую карту компетенций на уровне отдела.

**Оси:**
- Y — сотрудники отдела
- X — компетенции (общие + специфичные для отдела)
- Цвет ячейки — уровень (градиент от красного (0) до зелёного (4))

**Функциональность:**
- Фильтрация по категории компетенций
- Сортировка: по ФИО, по среднему уровню, по конкретной компетенции
- Отображение среднего уровня по отделу в нижней строке
- Клик по ячейке — детализация оценки (источники, история)

**Правила:**
- Доступна: TL — своя команда; Head/Admin — все отделы; HR — все (read-only)
- Обезличенный режим для HR (вместо ФИО — идентификаторы)

#### FR-032: История оценок

Система должна хранить и отображать полную историю оценок.

**Функциональность:**
- Таблица: кампания, дата, компетенция, оценка (self/peers/TL/Head/агрегированная)
- График динамики: ось X — кампании (время), ось Y — уровень (0–4)
- Фильтрация по компетенции, категории, периоду
- Сравнение двух кампаний (side-by-side)

**Правила:**
- История хранится бессрочно
- Employee видит свою историю с агрегированными оценками
- TL/Head/Admin видят полную детализацию (все источники)

#### FR-033: Gap-анализ

Система должна автоматически рассчитывать и визуализировать разрыв между текущим и целевым уровнем компетенций.

**Функциональность:**
- Таблица: компетенция | текущий уровень | целевой уровень | разрыв (gap)
- Цветовая индикация: зелёный (gap = 0), жёлтый (gap = 1), оранжевый (gap = 2), красный (gap >= 3)
- Сортировка по размеру gap (от большего к меньшему)
- Рекомендация обучающих ресурсов для компетенций с gap > 0 (из FR-013)

**Правила:**
- Gap = целевой уровень − текущий уровень (если < 0, то gap = 0, компетенция превышает целевой)
- Gap-анализ доступен: Employee — свой; TL — команда; Head/Admin — все

#### FR-034: Экспорт отчётов

Система должна поддерживать экспорт данных и отчётов.

**Форматы:**
- **Excel (XLSX):** таблицы оценок, gap-анализ, списки сотрудников, матрица компетенций
- **PDF:** радарные диаграммы, тепловые карты, графики динамики, сводный отчёт по сотруднику

**Типы отчётов:**
- Индивидуальный отчёт сотрудника (все компетенции, история, gap-анализ)
- Отчёт по отделу (тепловая карта, средние уровни, top/bottom компетенции)
- Отчёт по кампании (статистика прохождения, агрегированные результаты)
- Сводный отчёт по управлению (для Head)

**Правила:**
- Экспорт доступен ролям TL, Head, HR, Admin
- Employee может экспортировать свой индивидуальный отчёт
- PDF содержит дату генерации и водяной знак «Конфиденциально»

#### FR-035: Дашборд руководителя

Система должна предоставлять сводный дашборд для руководителя управления.

**Виджеты дашборда:**
- Общее количество сотрудников по отделам (бар-чарт)
- Средний уровень компетенций по отделам (бар-чарт)
- Статус текущей кампании оценки (прогресс-бар: сколько % оценок завершено)
- Top-5 компетенций с наибольшим gap по управлению
- Top-5 сотрудников с наибольшим ростом за последний цикл
- Распределение сотрудников по уровням (гистограмма)

**Правила:**
- Дашборд доступен: Head, Admin, HR
- Данные обновляются при каждом открытии (без кэширования)

### 2.5 Модуль "Карьерные треки"

#### FR-040: Определение карьерных путей

Система должна позволять определять карьерные пути между отделами и должностями.

**Атрибуты карьерного пути:**
- Название (например, «Первая линия → Дежурная смена»)
- Отдел/должность — источник
- Отдел/должность — цель
- Описание
- Среднее время перехода (в месяцах)

**Правила:**
- Карьерный путь может быть последовательным (1 → 2 → 3) или прямым (1 → 3)
- Admin и Head могут создавать и редактировать карьерные пути
- Сотрудник может видеть доступные ему карьерные пути

#### FR-041: Требуемые компетенции для перехода

Для каждого карьерного пути определяется набор компетенций и минимальных уровней, необходимых для перехода.

**Правила:**
- Набор компетенций берётся из целевого профиля целевого отдела/должности
- Admin/Head могут переопределить требования для конкретного пути
- Требуемые компетенции отображаются сотруднику при просмотре карьерного пути

#### FR-042: Визуализация карьерного пути

Система должна визуализировать карьерные пути.

**Формат визуализации:**
- Граф/дерево: узлы — отделы/должности, рёбра — карьерные пути
- При наведении на ребро — список требуемых компетенций и индикатор готовности
- Текущая позиция сотрудника выделена на графе

**Правила:**
- Визуализация доступна: Employee — свои возможные пути; TL/Head/Admin — все пути

#### FR-043: Gap-анализ для целевой роли

Система должна показывать gap-анализ сотрудника относительно целевой роли в другом отделе.

**Функциональность:**
- Таблица и радар: текущие компетенции vs требования целевой роли
- Список недостающих компетенций (которых нет в текущем профиле)
- Рекомендация обучающих ресурсов для закрытия gap

**Правила:**
- Расчёт аналогичен FR-033, но с профилем целевой роли
- Доступно: Employee — для себя; TL/Head/Admin — для любого сотрудника

#### FR-044: Индикатор готовности к переходу

Система должна отображать процент готовности сотрудника к переходу на целевую роль.

**Формула:**

```
Готовность (%) = (Σ min(текущий_уровень, целевой_уровень) / Σ целевой_уровень) × 100
```

**Правила:**
- Индикатор отображается на карточке карьерного пути
- Цветовая индикация: < 50% — красный, 50–79% — жёлтый, >= 80% — зелёный
- При готовности >= 80% TL и Head получают уведомление (если включено)

### 2.6 Модуль "Планы развития"

#### FR-050: Создание индивидуального плана развития (IDP)

Система должна позволять создавать индивидуальные планы развития.

**Атрибуты IDP:**
- Название
- Сотрудник (владелец)
- Период (дата начала — дата окончания)
- Цели (список)
- Статус: Черновик | Активен | Завершён

**Правила:**
- IDP создаёт сотрудник (для себя) или TL (для членов команды)
- IDP согласуется тимлидом (статус Черновик → Активен)
- Один сотрудник может иметь один активный IDP

#### FR-051: Привязка целей IDP к компетенциям

Каждая цель IDP должна быть связана с конкретной компетенцией.

**Атрибуты цели:**
- Описание цели
- Компетенция (из каталога)
- Текущий уровень (на момент создания)
- Целевой уровень
- Дедлайн
- Статус: Запланировано | В процессе | Завершено

**Правила:**
- Целевой уровень должен быть выше текущего
- Цель считается завершённой вручную (сотрудником) или автоматически (по результатам следующей оценки)
- При создании IDP система предлагает цели на основе gap-анализа (FR-033)

#### FR-052: Привязка обучающих ресурсов к целям IDP

Система должна связывать обучающие ресурсы с целями IDP.

**Правила:**
- При создании цели система автоматически предлагает ресурсы из каталога (FR-013), релевантные компетенции и целевому уровню
- Сотрудник может добавить собственные ресурсы к цели
- TL может рекомендовать ресурсы для конкретной цели

#### FR-053: Статусы целей

Жизненный цикл цели IDP:

```
Запланировано → В процессе → Завершено
```

**Правила:**
- Переход Запланировано → В процессе: сотрудник начинает работу
- Переход В процессе → Завершено: сотрудник отмечает завершение + подтверждение TL
- Возможен возврат: В процессе → Запланировано (при переоценке приоритетов)

#### FR-054: Трекинг прогресса IDP

Система должна отображать прогресс выполнения IDP.

**Функциональность:**
- Прогресс-бар: % завершённых целей от общего количества
- Таймлайн: дедлайны целей на временной шкале
- Уведомления о приближающихся дедлайнах (за 7 дней)
- Сводка для TL: прогресс IDP по всем сотрудникам команды

**Правила:**
- Прогресс обновляется при изменении статуса цели
- TL видит прогресс своей команды, Head — всех

### 2.7 Модуль "Аутентификация и авторизация"

#### FR-060: Регистрация и логин

Система должна поддерживать аутентификацию через email и пароль.

**Регистрация:**
- Поля: email, пароль, подтверждение пароля
- Email должен быть уникальным
- Пароль: минимум 8 символов, минимум 1 заглавная, 1 строчная, 1 цифра

**Логин:**
- Поля: email, пароль
- Блокировка аккаунта после 5 неудачных попыток (на 15 минут)
- Логин возможен только для активных сотрудников

**Правила:**
- Регистрация нового пользователя привязывает его к существующей карточке сотрудника (по email)
- Первый зарегистрированный пользователь получает роль Admin (bootstrap)

#### FR-061: JWT-токены

Система должна использовать JWT для аутентификации API-запросов.

**Access Token:**
- Срок действия: 30 минут
- Содержит: user_id, email, role
- Передаётся в заголовке Authorization: Bearer {token}

**Refresh Token:**
- Срок действия: 7 дней
- Хранится в httpOnly cookie
- Используется для получения нового access token

**Правила:**
- При протухании access token клиент автоматически использует refresh token
- При протухании refresh token — перенаправление на логин
- Logout инвалидирует refresh token на сервере

#### FR-062: RBAC (Role-Based Access Control)

Система должна разграничивать доступ на основе ролей.

**Матрица доступа:**

| Ресурс / Действие | Admin | Head | TL | HR | Employee |
|-------------------|-------|------|-----|-----|----------|
| Управление отделами | CRUD | Read | Read | Read | Read |
| Управление командами | CRUD | CRUD | Read (своя) | Read | Read (своя) |
| Управление сотрудниками | CRUD | CRUD | Update (своя команда) | Read | Read (свой) |
| Назначение ролей | CRUD | — | — | — | — |
| Каталог компетенций | CRUD | CRUD | Read | Read | Read |
| Целевые профили | CRUD | CRUD | Read | Read | Read |
| Кампании оценки | CRUD | CRUD | Read (своя команда) | Read | Read (свои) |
| Оценивание | All | All | Своя команда | — | Самооценка |
| Аналитика / Отчёты | All | All | Своя команда | All (read, вкл. индивидуальные оценки) | Свой профиль |
| Карьерные треки | CRUD | CRUD | Read | Read | Read (свои) |
| IDP | All | All | Своя команда | Read | Свой |
| Импорт данных | Yes | — | — | Yes | — |
| Экспорт отчётов | Yes | Yes | Своя команда | Yes | Свой |
| Настройки системы | Yes | — | — | — | — |

#### FR-063: Изоляция данных по ролям

Система должна гарантировать, что пользователь видит только данные, соответствующие его роли и области ответственности.

**Правила:**
- Employee: видит только свой профиль, свои оценки (агрегированные), свой IDP
- TL: видит свою команду полностью (все оценки, включая исходные)
- Head: видит все отделы и всех сотрудников
- HR: видит все персональные данные (ФИО, баллы) и индивидуальные оценки по всему управлению; не может редактировать IDP
- Admin: полный доступ без ограничений
- Все API-эндпоинты проверяют роль и scope перед выдачей данных
- В логах фиксируются попытки несанкционированного доступа

## 3. Нефункциональные требования

### Производительность

#### NFR-001: Время ответа API

- p95 время ответа API < 500 мс для любого эндпоинта
- p99 время ответа API < 1000 мс
- Время генерации отчёта (PDF/Excel) < 5 секунд

#### NFR-002: Одновременные пользователи

- Система должна поддерживать минимум 50 одновременных пользователей без деградации производительности
- Пиковая нагрузка (период оценки): до 50 пользователей одновременно заполняют формы оценки

#### NFR-003: Доступность

- Uptime не менее 99.5% (за месяц)
- Плановое окно обслуживания: не более 2 часов в месяц, в нерабочее время
- Graceful degradation: при недоступности Telegram-бота основная функциональность работает

### Дизайн и UX

#### NFR-004: Responsive дизайн

- Desktop-first подход
- Минимальное поддерживаемое разрешение: 1280x720
- Адаптация под планшеты (1024px) — опционально, для MVP не обязательно
- Мобильная версия не входит в MVP

#### NFR-010: Доступность (accessibility)

- Контрастность текста соответствует WCAG 2.1 AA
- Все интерактивные элементы доступны с клавиатуры
- Все изображения и графики имеют alt-текст

### Безопасность

#### NFR-005: HTTPS

- Все коммуникации между клиентом и сервером — только по HTTPS
- HTTP-запросы перенаправляются на HTTPS (301 redirect)
- Минимальная версия TLS: 1.2

#### NFR-006: Хэширование паролей

- Пароли хранятся в БД только в хэшированном виде
- Алгоритм: bcrypt с cost factor >= 12
- Пароли в открытом виде не логируются и не передаются в ответах API

#### NFR-007: OWASP Top 10

Система должна быть защищена от основных угроз OWASP Top 10:

- SQL Injection — использование ORM / параметризованных запросов
- XSS — экранирование вывода, Content-Security-Policy
- CSRF — использование CSRF-токенов для мутирующих операций
- Broken Authentication — JWT + rate limiting + account lockout
- Sensitive Data Exposure — HTTPS + шифрование чувствительных данных
- Security Misconfiguration — hardened конфигурация, отключение debug в production

#### NFR-011: Аудит безопасности

- Все действия с данными логируются (кто, что, когда)
- Логи не содержат чувствительных данных (паролей, токенов)
- Доступ к логам только для Admin

### Надёжность

#### NFR-008: Резервное копирование

- Автоматические бэкапы базы данных ежедневно
- Хранение бэкапов: минимум 30 дней
- Recovery Point Objective (RPO): 24 часа
- Recovery Time Objective (RTO): 4 часа
- Тестирование восстановления из бэкапа: ежемесячно

#### NFR-009: Логирование

- Логирование всех действий пользователей (аудит-трейл)
- Уровни логирования: DEBUG, INFO, WARNING, ERROR, CRITICAL
- Формат логов: JSON (для парсинга и анализа)
- Ротация логов: ежедневно, хранение 90 дней
- Логирование входящих HTTP-запросов: метод, URL, статус, время ответа, user_id

### Развёртывание

#### NFR-012: Cloud-деплой

- Развёртывание в облачной среде
- Контейнеризация (Docker)
- CI/CD pipeline для автоматического развёртывания
- Раздельные среды: dev, staging, production

#### NFR-013: Миграции БД

- Все изменения схемы БД через миграции (Alembic или аналог)
- Миграции идемпотентны и обратимы (up/down)
- Автоматическое применение миграций при деплое

## 4. User Stories

### Сотрудник (Employee)

**US-001:** Как сотрудник, я хочу видеть свой профиль с текущими оценками на радарной диаграмме, чтобы понимать свой текущий уровень компетенций.

**US-002:** Как сотрудник, я хочу пройти самооценку по всем компетенциям моего профиля, чтобы зафиксировать своё видение собственных навыков.

**US-003:** Как сотрудник, я хочу видеть gap-анализ между моими текущими оценками и целевым профилем, чтобы понимать, над чем работать.

**US-004:** Как сотрудник, я хочу видеть обучающие ресурсы, привязанные к компетенциям с gap > 0, чтобы знать, как закрыть разрыв.

**US-005:** Как сотрудник, я хочу видеть доступные карьерные пути и свой процент готовности к переходу, чтобы планировать развитие.

**US-006:** Как сотрудник, я хочу создать индивидуальный план развития с конкретными целями, чтобы структурировать своё обучение.

**US-007:** Как сотрудник, я хочу получать уведомления в Telegram о начале и дедлайне кампании оценки, чтобы не пропустить сроки.

**US-008:** Как сотрудник, я хочу видеть историю своих оценок и график динамики, чтобы отслеживать прогресс.

**US-009:** Как сотрудник, я хочу экспортировать свой индивидуальный отчёт в PDF, чтобы использовать его при обсуждении с руководителем.

### Тимлид (Team Lead)

**US-010:** Как тимлид, я хочу видеть тепловую карту компетенций моей команды, чтобы выявить сильные и слабые стороны.

**US-011:** Как тимлид, я хочу оценить каждого сотрудника команды в рамках кампании 360°, чтобы зафиксировать своё мнение о их уровне.

**US-012:** Как тимлид, я хочу назначить peer-оценщиков для каждого сотрудника, чтобы обеспечить объективность оценки.

**US-013:** Как тимлид, я хочу видеть прогресс IDP каждого сотрудника команды, чтобы контролировать их развитие.

**US-014:** Как тимлид, я хочу экспортировать отчёт по команде в Excel, чтобы использовать данные для планирования.

**US-015:** Как тимлид, я хочу видеть gap-анализ по каждому сотруднику, чтобы определить приоритеты обучения.

### Руководитель (Department Head)

**US-020:** Как руководитель, я хочу видеть сводный дашборд по всему управлению, чтобы иметь общую картину компетенций.

**US-021:** Как руководитель, я хочу создать и запустить кампанию оценки 360°, чтобы провести регулярную оценку сотрудников.

**US-022:** Как руководитель, я хочу видеть тепловую карту по каждому отделу, чтобы сравнивать уровни компетенций между отделами.

**US-023:** Как руководитель, я хочу определять карьерные пути между отделами, чтобы сотрудники понимали возможности роста.

**US-024:** Как руководитель, я хочу настроить целевые профили для каждой должности, чтобы формализовать ожидания по компетенциям.

**US-025:** Как руководитель, я хочу видеть Top-5 компетенций с наибольшим gap, чтобы планировать обучающие мероприятия.

### HR

**US-030:** Как HR, я хочу импортировать список сотрудников из Excel, чтобы быстро наполнить систему данными.

**US-031:** Как HR, я хочу генерировать сводные отчёты по компетенциям для руководства, чтобы поддерживать принятие решений.

**US-032:** Как HR, я хочу видеть аналитику по сотрудникам в обезличенном виде, чтобы оценивать общие тенденции.

**US-033:** Как HR, я хочу экспортировать данные о компетенциях в Excel, чтобы интегрировать их с другими HR-системами.

### Администратор (Admin)

**US-040:** Как администратор, я хочу управлять каталогом компетенций (CRUD), чтобы поддерживать актуальность матрицы.

## 6. Критерии приёмки (Acceptance Criteria)

### AC-001: Профиль сотрудника

- [ ] При открытии профиля отображаются все поля: ФИО, должность, отдел, команда, email, Telegram
- [ ] Радарная диаграмма отображает текущие оценки и целевой профиль
- [ ] Gap-анализ корректно рассчитывает разницу между текущим и целевым уровнем
- [ ] Фильтр по категориям компетенций работает на радаре и в таблице
- [ ] История оценок отображает данные за все прошедшие кампании
- [ ] Кнопки экспорта генерируют корректные файлы PDF и Excel

### AC-002: Кампания оценки 360°

- [ ] Admin/Head может создать кампанию с названием, датами и участниками
- [ ] При активации кампании участники получают уведомление в Telegram
- [ ] Сотрудник может пройти самооценку: выбрать уровень 0-4 для каждой компетенции
- [ ] Самооценку можно сохранить как черновик и завершить позже
- [ ] После отправки самооценка не подлежит изменению
- [ ] TL может назначить 2-5 peer-оценщиков для каждого сотрудника
- [ ] Peer-оценки анонимны для оцениваемого (отображаются только агрегированные)
- [ ] Агрегация корректно вычисляется по выбранному методу (среднее/медиана/взвешенное)
- [ ] Статус кампании корректно меняется: Черновик → Активна → Завершена → Архив
- [ ] Напоминания отправляются за 3 дня и за 1 день до дедлайна
- [ ] При завершении кампании незаполненные оценки помечаются как «не предоставлена»

### AC-003: Тепловая карта

- [ ] Тепловая карта отображает матрицу: сотрудники x компетенции
- [ ] Ячейки окрашены по градиенту: красный (0) → зелёный (4)
- [ ] Нижняя строка содержит средние значения по каждой компетенции
- [ ] Фильтрация по категории компетенций работает корректно
- [ ] Сортировка работает: по ФИО, по среднему баллу, по конкретной компетенции
- [ ] Клик по ячейке открывает детализацию: источники оценки, история
- [ ] Для роли HR ФИО заменены на обезличенные идентификаторы

### AC-004: Каталог компетенций

- [ ] Admin может создать, отредактировать, архивировать компетенцию
- [ ] Компетенции разделяются на общие и специфичные
- [ ] Специфичные компетенции привязаны к одному или нескольким отделам
- [ ] Для каждого уровня (0-4) можно задать описание критериев
- [ ] Обучающие ресурсы привязываются к компетенциям
- [ ] Импорт из CSV/Excel: предварительный просмотр, валидация, построчные ошибки

### AC-005: Карьерные треки

- [ ] Определены карьерные пути между отделами с требуемыми компетенциями
- [ ] Визуальный граф отображает все возможные пути
- [ ] Текущая позиция сотрудника выделена на графе
- [ ] Индикатор готовности (%) рассчитывается по формуле и отображается корректно
- [ ] Gap-анализ для целевой роли показывает разницу между текущими и требуемыми компетенциями
- [ ] Рекомендуются обучающие ресурсы для закрытия gap

### AC-006: Аутентификация и авторизация

- [ ] Регистрация: email уникален, пароль соответствует требованиям сложности
- [ ] Логин: корректные credentials → JWT-токены, неверные → ошибка
- [ ] Блокировка после 5 неудачных попыток на 15 минут
- [ ] Access token истекает через 30 минут, refresh token — через 7 дней
- [ ] Каждый API-эндпоинт проверяет роль пользователя
- [ ] Employee не может получить доступ к данным других сотрудников через API
- [ ] TL видит только свою команду
- [ ] Head видит все отделы
- [ ] HR видит агрегированные данные без детализации отдельных оценок

### AC-007: Импорт данных

- [ ] CSV и XLSX файлы успешно парсятся
- [ ] Перед импортом отображается предварительный просмотр данных
- [ ] Ошибки валидации отображаются с номером строки и описанием
- [ ] Дубликаты по email обнаруживаются и предлагается обновление (с подтверждением)
- [ ] После успешного импорта данные сразу доступны в системе

### AC-008: Индивидуальный план развития (IDP)

- [ ] Сотрудник может создать IDP с целями, привязанными к компетенциям
- [ ] Система предлагает цели на основе gap-анализа
- [ ] К каждой цели можно привязать обучающие ресурсы
- [ ] Статусы целей корректно переключаются: Запланировано → В процессе → Завершено
- [ ] Прогресс-бар отображает % завершённых целей
- [ ] TL видит прогресс IDP всех сотрудников команды
- [ ] Уведомления о дедлайнах отправляются за 7 дней

### AC-009: Экспорт отчётов

- [ ] Excel-файл содержит корректные данные (таблицы, числа, форматирование)
- [ ] PDF содержит графики (радар, тепловая карта), дату генерации и водяной знак
- [ ] Индивидуальный отчёт доступен Employee для экспорта

## Приложение B: Весовые коэффициенты для взвешенной агрегации (по умолчанию)

| Источник оценки | Вес по умолчанию | Обоснование |
|-----------------|------------------|-------------|
| Руководитель управления/отдела (Head) | 0.35 | Стратегический взгляд, оценка в контексте отдела и управления |
| Тимлид (TL) | 0.30 | Наиболее близкий наблюдатель повседневной работы |
| Самооценка | 0.20 | Субъективна, но важна для самосознания |
| Peers (коллеги) | 0.15 | Горизонтальная обратная связь, нивелируется усреднением нескольких оценок |

Формула: `final_score = head_score × 0.35 + tl_score × 0.30 + self_score × 0.20 + peer_avg × 0.15`

Сумма весов = 1.00. Веса настраиваются Admin. При отсутствии какого-либо источника веса перераспределяются пропорционально.

**Пример перераспределения (без оценки Head):**
- TL: 0.30 / 0.65 ≈ 0.4615
- Самооценка: 0.20 / 0.65 ≈ 0.3077
- Peers: 0.15 / 0.65 ≈ 0.2308
