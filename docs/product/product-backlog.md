# Product Backlog — Matrix

**Проект:** Matrix — веб-приложение матрицы компетенций
**Заказчик:** Управление сопровождения портала динамической инфраструктуры Сбер
**Масштаб:** 5 отделов, 10–50 сотрудников
**Команда:** Solo developer + AI
**Приоритизация:** MoSCoW (Must / Should / Could / Won't)
**Спринты:** 2 недели, ~20 SP на спринт (Solo developer + AI)
**Дата создания:** 2026-03-24
**Версия:** 1.0

## Роли пользователей

| Роль | Описание |
|------|----------|
| **Admin** | Администратор системы — полный доступ к управлению |
| **Head** | Руководитель управления — видит все отделы и сотрудников |
| **TL** | Тимлид — управляет конкретной командой внутри отдела |
| **HR** | HR-специалист — доступ к отчётам и аналитике |
| **Employee** | Сотрудник — просмотр своего профиля, самооценка |
| **System** | Система — автоматические процессы |
| **Developer** | Разработчик — инфраструктура и DevOps |

## Definition of Done

Пользовательская история считается завершённой, когда выполнены **все** критерии:

### Код

- [ ] Код написан, прошёл code review (self-review + AI review)
- [ ] Нет линтер-ошибок (ruff для Python, ESLint для TypeScript)
- [ ] Нет TODO/FIXME в коде, относящихся к этой истории

### Тесты

- [ ] Unit-тесты написаны и проходят (backend: pytest, frontend: vitest)
- [ ] Integration-тесты для API-эндпоинтов написаны и проходят
- [ ] Покрытие тестами для нового кода >= 80%
- [ ] Все тесты проходят в CI

### Функциональность

- [ ] Критерии приёмки (Acceptance Criteria) выполнены и проверены
- [ ] Функциональность работает в Docker-окружении
- [ ] Ролевая модель доступа соблюдена (RBAC)
- [ ] Нет регрессий в существующей функциональности

### Документация

- [ ] API-эндпоинты описаны в OpenAPI (автогенерация FastAPI)
- [ ] Миграции БД созданы и применяются корректно

### Деплой

- [ ] CI pipeline проходит без ошибок
- [ ] Docker-образ собирается успешно
- [ ] Feature branch слита в develop

## Формат критериев приёмки

Каждая User Story должна содержать критерии приёмки в формате **Given / When / Then** (Gherkin-стиль). Это обеспечивает однозначность требований и возможность автоматизации тестирования.

### Шаблон

```
### US-XXX: [Название]

**Как** [роль],
**я хочу** [действие],
**чтобы** [цель/ценность].

**Критерии приёмки:**

**AC-1:** [Название критерия]
- **Given** [предусловие]
- **When** [действие пользователя]
- **Then** [ожидаемый результат]

**AC-2:** [Название критерия]
- **Given** [предусловие]
- **When** [действие пользователя]
- **Then** [ожидаемый результат]

**Негативный сценарий:**
- **Given** [предусловие]
- **When** [некорректное действие]
- **Then** [сообщение об ошибке / блокировка]
```

### Пример

```
### US-002: Вход по email/паролю

**Как** сотрудник,
**я хочу** войти в систему по email и паролю,
**чтобы** получить доступ к своему профилю и функциям системы.

**Критерии приёмки:**

**AC-1:** Успешный вход
- **Given** сотрудник зарегистрирован и имеет активную учётную запись
- **When** он вводит корректный email и пароль и нажимает "Войти"
- **Then** система перенаправляет на главную страницу, JWT-токен сохранён

**AC-2:** Отображение имени пользователя
- **Given** сотрудник успешно вошёл в систему
- **When** отображается шапка приложения
- **Then** в шапке видны ФИО и роль пользователя

**Негативный сценарий:**
- **Given** сотрудник вводит неверный пароль
- **When** он нажимает "Войти"
- **Then** система показывает ошибку "Неверный email или пароль" (без указания, что именно неверно)
```

### Рекомендации по написанию AC

1. **Один критерий = одна проверка.** Не объединяйте несколько проверок в один AC.
2. **Всегда включайте негативный сценарий.** Как система ведёт себя при ошибке?
3. **Указывайте конкретные значения**, а не абстрактные ("показывает ошибку" -> "показывает текст 'Неверный email или пароль'").
4. **Учитывайте RBAC.** Если функция доступна только определённым ролям, это отдельный AC.
5. **Формулируйте тестируемо.** Каждый AC должен быть проверяем — вручную или автоматически.

## Приложение: Приоритизация MoSCoW

| Категория | Определение | Доля |
|-----------|-------------|------|
| **Must Have** | Без этого продукт не имеет смысла. MVP-функциональность. Блокирует релиз. | ~51% SP |
| **Should Have** | Важно, но MVP может работать без этого. Реализуется сразу после Must. | ~35% SP |
| **Could Have** | Желательно, но можно отложить. Реализуется при наличии ресурсов. | ~13% SP |
| **Won't Have** | Не в этом релизе. Фиксируется для будущих версий. | ~1% SP |

**Правило:** Must Have не должен превышать 60% от общего объёма SP. Текущее распределение (51%) оставляет буфер для непредвиденных задач и технического долга.
