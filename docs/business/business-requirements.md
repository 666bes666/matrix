## 1. Описание проекта

### 1.1. Предпосылки

Управление сопровождения портала динамической инфраструктуры Сбер включает 5 отделов с численностью от 10 до 50 сотрудников. На текущий момент оценка компетенций сотрудников не ведётся: отсутствует единая система учёта навыков, нет формализованных карьерных треков, планирование развития происходит ситуативно. Руководители отделов не имеют инструмента для объективного сравнения уровня подготовки сотрудников и выявления пробелов в компетенциях.

### 1.2. Проблемы текущего состояния

- Отсутствие единого источника данных о компетенциях сотрудников
- Невозможность объективно сравнить уровни подготовки внутри и между отделами
- Нет формализованных критериев для карьерного роста и перехода между отделами
- Планирование обучения не привязано к реальным пробелам в навыках
- Риски потери квалифицированных кадров из-за непрозрачности карьерных перспектив
- Высокие трудозатраты на ручной сбор и анализ информации о навыках

### 1.3. Предлагаемое решение

Matrix — веб-приложение матрицы компетенций, обеспечивающее полный цикл оценки: от формирования каталога компетенций и проведения оценки 360° до визуализации результатов, построения карьерных треков и формирования индивидуальных планов развития.

## 2. Цели проекта

### 2.1. Стратегические цели

| ID | Цель | Измеримый результат |
|----|------|---------------------|
| G-001 | Создать единую платформу оценки компетенций | Все 5 отделов используют систему как единственный источник данных о компетенциях |
| G-002 | Обеспечить прозрачные карьерные треки | Для каждого отдела сформирован целевой профиль компетенций; сотрудники видят разрыв между текущим и целевым уровнем |
| G-003 | Снизить время на оценку и планирование развития | Сокращение трудозатрат на проведение оценки не менее чем на 50% по сравнению с ручным процессом |
| G-004 | Повысить retention через прозрачность роста | Сотрудники имеют доступ к карьерным трекам, индивидуальным планам развития и динамике собственного роста |
| G-005 | Создать основу для data-driven решений в HR | Руководители и HR принимают решения о ротации, обучении и продвижении на основе объективных данных |

### 2.2. Тактические цели

- Провести первый полный цикл оценки 360° для всех 5 отделов в течение 3 месяцев после запуска
- Сформировать каталог компетенций, покрывающий все 4 категории: Hard Skills, Soft Skills, Процессы, Domain Knowledge
- Обеспечить доступность системы через веб-интерфейс без необходимости установки ПО на рабочие станции
- Интегрировать Telegram-уведомления для оперативного информирования участников оценки

## 3. Стейкхолдеры

### 3.1. Карта стейкхолдеров

| Роль | Описание | Интересы | Влияние | Вовлечённость |

| Тимлиды | Непосредственные руководители команд внутри отделов | Инструмент оценки подчинённых; понимание сильных/слабых сторон команды; планирование задач с учётом компетенций | Среднее | Активное — участие в оценке 360°, формирование ИПР |

| Сотрудники (10-50 чел.) | Объекты и участники оценки | Понимание своего уровня; прозрачность требований для роста; доступ к обучающим материалам; справедливость оценки | Низкое | Регулярное — прохождение самооценки, просмотр результатов и ИПР |

| Роль в системе | Соответствие организационной роли | Ключевые полномочия |
|----------------|-----------------------------------|---------------------|
| Admin | Администратор системы / HR-менеджер | Полный доступ: управление пользователями, каталогом компетенций, настройками системы, циклами оценки |
| Department Head | Руководитель отдела | Управление сотрудниками отдела, просмотр всех оценок отдела, формирование целевых профилей, утверждение ИПР |

| Employee | Сотрудник | Самооценка, оценка коллег (peer review), просмотр собственных результатов и ИПР |

## 4. Организационная структура

### 4.1. Отделы управления

Отделы перечислены в порядке от менее к более квалифицированному. Этот порядок определяет логику карьерных треков.

| № | Отдел | Зона ответственности | Ключевые компетенции |
|---|-------|---------------------|---------------------|
| 1 | Первая линия сопровождения | Базовая поддержка пользователей, управление квотами, обработка известных багов по базе знаний | Работа с тикетами, базовые знания платформы, коммуникация с пользователями |
| 2 | Дежурная смена | Troubleshooting, работа с Ansible, управление vCloud Director, обработка инцидентов | Ansible, vCloud Director, диагностика инцидентов, работа в нерабочее время |
| 3 | Бизнес-логика | Бизнес-логика портала, управление ресурсами платформы | Понимание бизнес-процессов портала, управление ресурсами, взаимодействие с разработкой |
| 4 | Вторая линия (SRE/DevOps) | Работа с логами, метриками, кодом, базами данных | Мониторинг (Grafana, ELK), скриптинг, SQL, работа с кодом платформы, SRE-практики |
| 5 | Сопровождение Jenkins CDP | Автоматизация CI/CD-процессов | Jenkins, CI/CD-пайплайны, Groovy, автоматизация деплоя |

### 4.2. Карьерный трек между отделами

Типичная траектория роста: Первая линия -> Дежурная смена -> Бизнес-логика / SRE/DevOps -> Сопровождение Jenkins CDP. Переход возможен при достижении целевого профиля компетенций следующего отдела.

| BR-002 | Система должна обеспечивать управление профилями сотрудников: ФИО, должность, отдел, команда, дата приёма, контактные данные, Telegram-username | Высокий | Необходимо для идентификации сотрудников и отправки уведомлений |
| BR-003 | Система должна поддерживать перевод сотрудника между отделами с сохранением истории оценок | Средний | Карьерные треки предполагают переход между отделами |

### 5.2. Каталог компетенций

| ID | Требование | Приоритет | Обоснование |
|----|-----------|-----------|-------------|
| BR-004 | Система должна обеспечивать ведение каталога компетенций с категоризацией по четырём типам: Hard Skills, Soft Skills, Процессы, Domain Knowledge | Высокий | Структурированный каталог — основа для проведения оценки |
| BR-005 | Каждая компетенция должна иметь описание, категорию, привязку к отделам и 5 уровней владения (0 — нет навыка, 1 — базовый, 2 — средний, 3 — продвинутый, 4 — экспертный) с текстовым описанием каждого уровня | Высокий | Единая шкала обеспечивает объективность и сопоставимость оценок |
| BR-006 | Система должна поддерживать формирование целевых профилей компетенций для каждого отдела и должности — набор компетенций с минимально необходимыми уровнями | Высокий | Целевой профиль необходим для gap-анализа и построения карьерных треков |
| BR-007 | Администратор и руководитель отдела должны иметь возможность добавлять, редактировать и деактивировать компетенции в каталоге | Средний | Каталог компетенций должен развиваться вместе с организацией |

### 5.3. Оценка 360°

| ID | Требование | Приоритет | Обоснование |
|----|-----------|-----------|-------------|
| BR-008 | Система должна обеспечивать проведение оценки 360° с четырьмя типами оценщиков: самооценка, оценка коллегами (peers), оценка тимлидом, оценка руководителем отдела | Высокий | Методика 360° даёт наиболее объективную картину компетенций |
| BR-009 | Оценка должна проводиться циклами с настраиваемой периодичностью (по умолчанию — каждые 6 месяцев) | Высокий | Регулярность оценки обеспечивает отслеживание динамики развития |
| BR-010 | Система должна поддерживать настройку цикла оценки: дата начала, дата окончания, список участников, набор оцениваемых компетенций | Высокий | Гибкость настройки цикла необходима для адаптации под разные отделы |
| BR-011 | HR и руководитель отдела должны иметь возможность назначать оценщиков для каждого сотрудника в рамках цикла | Средний | Контроль за составом оценщиков обеспечивает релевантность оценки |
| BR-012 | Система должна отображать прогресс прохождения оценки в реальном времени: кто завершил, кто не начал, процент заполнения | Средний | Необходимо для контроля и своевременного напоминания |
| BR-013 | Итоговая оценка по каждой компетенции должна рассчитываться как взвешенное среднее оценок всех типов оценщиков с настраиваемыми весами | Средний | Разные типы оценщиков имеют разную значимость для итоговой оценки |
| BR-014 | Результаты оценки коллегами должны быть анонимными для оцениваемого сотрудника | Высокий | Анонимность повышает честность оценки |

### 5.4. Визуализация и аналитика

| ID | Требование | Приоритет | Обоснование |
|----|-----------|-----------|-------------|

|----|-----------|-----------|-------------|

## 7. Ограничения и допущения

### 7.1. Ограничения

| ID | Ограничение | Влияние |
|----|-------------|---------|
| C-001 | Разработка ведётся одним разработчиком с помощью AI-ассистента | Ограничивает скорость разработки; требует приоритизации функциональности и итеративного подхода (MVP -> расширение) |
| C-002 | Облачный деплой (без выделенной on-premise инфраструктуры) | Необходимо учитывать ограничения облачной платформы; не хранить данные, требующие размещения в контуре Сбера |
| C-003 | Аутентификация на базе локальной БД (без интеграции с корпоративным SSO/LDAP на первом этапе) | Требуется ручное создание учётных записей; в перспективе возможна интеграция |
| C-004 | Бюджет проекта ограничен стоимостью облачного хостинга | Влияет на выбор технологического стека — предпочтение open-source решениям |
| C-005 | Количество пользователей: 10-50 человек | Не требуется высоконагруженная архитектура; достаточно одного сервера приложений |

### 7.2. Допущения

| ID | Допущение | Риск при невыполнении |
|----|-----------|----------------------|
| A-001 | Руководители отделов готовы участвовать в формировании каталога компетенций и целевых профилей | Без участия руководителей каталог будет неполным, оценка — нерелевантной |
| A-002 | Сотрудники имеют доступ к веб-браузеру на рабочих станциях | Система недоступна для части сотрудников |
| A-003 | Сотрудники используют Telegram и готовы получать уведомления | Уведомления не достигнут части пользователей; необходим fallback |
| A-004 | Структура отделов и команд стабильна на горизонте 6-12 месяцев | Частая реорганизация потребует доработки системы |
| A-005 | Шкала оценки 0-4 достаточна для дифференциации уровней компетенций | Потребуется расширение шкалы и переоценка |
| A-006 | Данные о сотрудниках не содержат персональных данных, требующих соблюдения ФЗ-152 в полном объёме | Потребуется дополнительная работа по compliance |

### 7.3. Зависимости

| ID | Зависимость | Статус |
|----|-------------|--------|
| D-001 | Доступность Telegram Bot API для отправки уведомлений | Требует создания Telegram-бота и настройки webhook |
| D-002 | Наличие актуального списка сотрудников по отделам | Должен быть предоставлен HR до запуска |
| D-003 | Сформированный каталог компетенций (хотя бы для одного отдела) | Необходим для проведения первого цикла оценки |

## 9. Глоссарий

| Термин | Определение |
|--------|-------------|
| Компетенция | Измеримый навык или область знаний, необходимая для выполнения рабочих задач |
| Оценка 360° | Метод оценки, при котором сотрудник оценивается с нескольких сторон: самооценка, коллеги, непосредственный руководитель, вышестоящий руководитель |
| Целевой профиль | Набор компетенций с минимально необходимыми уровнями для определённой должности или отдела |
| Gap-анализ | Сравнение текущего уровня компетенций с целевым профилем для выявления пробелов |
| ИПР | Индивидуальный план развития — документ, фиксирующий цели развития сотрудника, необходимые действия и сроки |
| Цикл оценки | Период, в течение которого проводится оценка компетенций сотрудников (по умолчанию — 6 месяцев) |
| Карьерный трек | Визуализация возможных путей карьерного роста с указанием требуемых компетенций для перехода |
| Radar chart | Диаграмма-паук — тип визуализации для отображения нескольких количественных параметров на радиальных осях |
| Heatmap | Тепловая карта — матричная визуализация, использующая цвет для отображения интенсивности значений |
| NPS | Net Promoter Score — метрика лояльности, показывающая готовность пользователя рекомендовать продукт |
| Retention | Удержание сотрудников — метрика, обратная текучести персонала |
| SRE | Site Reliability Engineering — практики обеспечения надёжности и доступности систем |
| CDP | Continuous Delivery Pipeline — конвейер непрерывной доставки |
| MVP | Minimum Viable Product — минимально жизнеспособный продукт |
